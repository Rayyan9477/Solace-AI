## CRITICAL — Verified Open (3)

### C-01 remaining | Multi-worker JWT revocation broken
- **Location**: [src/solace_security/middleware.py:132](src/solace_security/middleware.py#L132)
- **Current state**: `_get_jwt_manager()` hardcodes `InMemoryTokenBlacklist()`
- **Why it matters**: In multi-worker deployments (uvicorn --workers N), token revocation in one worker does not propagate to others. Violates HIPAA audit requirements for session revocation.
- **Fix**: Replicate the pattern already used in [services/user-service/src/main.py:186-196](services/user-service/src/main.py#L186) — conditional `RedisTokenBlacklist` when Redis client is available, fall back to `InMemoryTokenBlacklist` only when `ENVIRONMENT=testing`.
- **Sprint**: 1 Day 1
- **TDD**: Write test asserting `_get_jwt_manager()` returns RedisTokenBlacklist when Redis URL set.

### NEW-03 | DiagnosisSession.messages not encrypted at rest
- **Location**: [src/solace_infrastructure/database/entities/diagnosis_entities.py:76](src/solace_infrastructure/database/entities/diagnosis_entities.py#L76)
- **Current state**: `__phi_fields__: ClassVar[list[str]] = ["summary"]` — `messages` column (JSONB conversation history) not in list
- **Additional blocker**: [src/solace_infrastructure/database/base_models.py:429](src/solace_infrastructure/database/base_models.py#L429) — `encrypt_phi_fields` only handles `isinstance(value, str)` types. Simply adding `messages` to the list won't encrypt it because JSONB is a list/dict type.
- **Why it matters**: Conversation history is PHI per HIPAA. Currently stored as plaintext JSONB.
- **Fix (2 parts)**:
  1. Enhance `encrypt_phi_fields` / `decrypt_phi_fields` in `base_models.py` to serialize list/dict via `json.dumps` before encrypting and `json.loads` after decrypting
  2. Add `"messages"` to `DiagnosisSession.__phi_fields__`
- **Sprint**: 1 Day 2
- **TDD**: Round-trip test storing a known-content list in `messages`, reading back, verifying plaintext matches + raw DB cell is ciphertext.

### NEW-04 | Hypothesis evidence not encrypted at rest
- **Location**: [src/solace_infrastructure/database/entities/diagnosis_entities.py:208](src/solace_infrastructure/database/entities/diagnosis_entities.py#L208)
- **Current state**: `class Hypothesis(ClinicalBase)` declares no `__phi_fields__`. Inherits default `[]`.
- **Fields to protect**: `supporting_evidence`, `contra_evidence`, `challenge_results`, `criteria_met`, `criteria_missing`, `hitop_dimensions` — all JSONB, all clinical content
- **Fix**: Declare `__phi_fields__: ClassVar[list[str]] = ["supporting_evidence", "contra_evidence", "challenge_results", "criteria_met", "criteria_missing"]` (hitop_dimensions is numeric scores, can stay plaintext). Depends on NEW-03 fix #1 (JSONB support in encrypt helpers).
- **Sprint**: 1 Day 2
- **TDD**: Similar round-trip test.

## HIGH — Verified Open (sampled from src/)

### NEW-05 | Service auth `Header()` dependency missing
- **Location**: [src/solace_security/service_auth.py:498-499](src/solace_security/service_auth.py#L498)
- **Current state**: `authorization: str | None = None` — no `Header()` wrapper
- **Why it matters**: FastAPI won't inject the Authorization header. Every service-auth-protected endpoint returns 401 "Missing Authorization header" even with a valid token.
- **Fix**: `from fastapi import Header; authorization: str | None = Header(None)`
- **Sprint**: 1 Day 1
- **TDD**: Write test hitting a service-auth-protected endpoint with valid `Authorization: Bearer <token>` header, expect 200.

### H-38 | Encryption key byte-length validation
- **Location**: [src/solace_security/encryption.py:86](src/solace_security/encryption.py#L86)
- **Current state**: `if len(key_value) != KEY_SIZE:` counts characters, not bytes
- **Why it matters**: Latent bug — works for ASCII keys (1 char = 1 byte) but fails for non-ASCII keys (e.g., if user provides a UTF-8 key with multi-byte chars)
- **Fix**: `if len(key_value.encode('utf-8')) != KEY_SIZE:` plus update error message to recommend `secrets.token_bytes(32)` instead of `token_urlsafe(32)[:32]`
- **Sprint**: 1 Day 2
- **TDD**: Test passing a 32-byte key containing a multi-byte char (e.g., `é` counts 1 char but 2 bytes); expect it to fail current validation, pass after fix.

### H-39 | PostgreSQL SSL enforcement disabled by default in prod
- **Location**: [src/solace_infrastructure/feature_flags.py:120](src/solace_infrastructure/feature_flags.py#L120)
- **Current state**: `enforce_database_ssl` feature flag defaults to False; SSL infra exists in `postgres.py:94-131` and `connection_manager.py:236-251`
- **Why it matters**: PHI transits unencrypted from service to DB on internal network. HIPAA-required in production.
- **Fix**: Flip default to True for production environment only. Write test validating SSL context is passed to asyncpg pool.
- **Sprint**: 1 Day 3

### NEW-01 | Audit HMAC digestmod — NOT A BUG
- **Location**: `src/solace_security/audit.py:152`
- **Original claim**: `hmac.new(key, msg, "sha256")` crashes at runtime
- **Verification (2026-04-12)**: `hmac.new()` has accepted a string digestmod since Python 3.4 (PEP 247). Verified: `python3 -c "import hmac; print(hmac.new(b'k', b'm', 'sha256').hexdigest())"` produces correct hash on Python 3.13. Same result as passing `hashlib.sha256` callable.
- **Status**: REMOVED from backlog. My April audit was incorrect.

- **Status**: REMOVED from backlog.
